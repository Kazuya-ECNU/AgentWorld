# Patch: LLM #3 按连通子图生成故事

**日期：** 2026-04-30
**涉及文件：**
- `src/agent_world/services/interaction_layer.py` — 核心理念变更
- `src/agent_world/services/graph_npc_engine.py` — 为 LLM #3 提供节点信息

---

## 一、背景：为什么需要改

### 1.1 原有设计的问题

LLM #3（叙事层）原先的输入只有两个来源：

1. **每个 NPC 的 `raw_intent`** — LLM #1 输出的自然语言计划
2. **去重后的 `EdgeResult` 列表** — 每条边一段故事

LLM #3 **看不到库存、看不到属性、看不到记忆、看不到 zone 信息**。它只能在两条信息之间讲故事——"老陈想卖皮毛"和"王老板想收皮毛"。

### 1.2 由此引发的三个问题

**问题 A：故事丢失。** LLM #3 输出 `【王老板↔老陈】`（合并写了一个联合故事），但 EdgeResult 是两条独立的 NPC×Zone 驻足边（`【王老板↔market】`、`【老陈↔market】`）。`_assign_stories` 匹配不上 → LLM #4 只看到"安静地待着"。  
*根因：LLM #3 的叙事单元（场景）和 EdgeResult 的单元（单条边）不在同一个抽象层级。*

**问题 B：信息匮乏导致叙事苍白。** LLM #3 不知道角色心情如何、最近经历了什么、身上带着什么，只能靠"想法"里的只言片语编故事。故事质量受限，LLM #4 能推理的素材也随之受限。

**问题 C：per-edge 与自然叙事矛盾。** 一条指令说"每条边单独写一段"；另一条又说"驻足 NPC 写 ta 在做什么"。LLM 在两人都在同一区域驻足时，自然想写"一个场景里两个人的互动"，但 prompt 格式要求拆成两段。

### 1.3 根本原因

**拓扑层（"谁和谁在一起"）和内容层（"他们身上有什么、在想什么"）纠缠不清。**  
老的 LLM #3 prompt 既负责承载分组（通过 per-edge 隐含"同一个场景"），又负责描述内容（想法）。两个职责混在一起，最终哪个都没做好。

---

## 二、改动思路

### 2.1 核心原则：拓扑与内容解耦

```
拓扑层（BFS 遍历 + is_leaf 标签）：决定"哪些节点构成同一个场景"
内容层（节点信息）：决定"这个故事讲什么"
                       ↓
LLM #3 只看内容，不知道拓扑
拓扑引擎只分组，不知道"场景"是什么语义
```

### 2.2 具体方案

#### a) 按连通子图分组 → 重构 LLM #3 的输入

从"给 LLM #3 传一个 EdgeResult 列表"改为"传 N 个场景包，每个场景包包含该连通子图内所有节点的信息"。

一个场景包的结构：

```
===== 场景: market =====                  ← 拓扑信息：这是一个子图

【场所】market                             ← zone 节点信息
【角色】老陈（FARMER）                     ← NPC 节点信息
  体力: 还行                               ← 数值→文字化（非结构化）
  饱腹: 有点饿                             ↑
  心情: 有点低落                           ↑
  最近经历: 之前用皮毛换了小麦...            ← 记忆（节点内容）
  性格: 平民、普通                          ← 性格特质（节点内容）
  想法: 想用皮毛换点金币...                  ← 计划（节点内容）

【角色】王老板（MERCHANT）
  体力: 精力充沛
  ...

【物品】                                    ← 子图中所有节点的库存
  · 老陈带着: 金币x35、皮毛x3、小麦x1
  · 王老板带着: 皮毛x2、金币x35

【交互】                                    ← 子图内的边
  · 老陈 在 market 驻足
  · 王老板 在 market 驻足
```

#### b) LLM #3 输出：每个场景一段 `【场景:xxx】`

```
【场景:market】清晨的阳光洒在市集的石板路上...老陈和王老板寒暄几句，
说起最近的生意都不太顺心，默契地开始讨价还价...
```

一段故事分配给该子图内所有边 → LLM #4 拿到完整场景叙事。

#### c) 未来可扩展：基于 BFS 的拓扑分组（is_leaf 标签）

当前按 zone 做字符串分组是过渡方案。真正的拓扑分组应该是：

- 从第一个 NPC 节点出发 BFS
- 遇到 `is_leaf=True` 的节点（item/object）→ 记录但不继续
- 遇到 `is_leaf=False` 的节点（npc/zone）→ 继续扩展
- 所有遍历到的非叶子节点构成一个连通分量

这样分组逻辑对语义完全透明——不需要知道"zone 是什么"，只需要知道"叶子还是非叶子"。

## 三、改动内容

### 3.1 `interaction_layer.py`

| 改动 | 从 | 到 |
|------|-----|------|
| `process()` | 接收 `edge_results` | 额外接收 `graph_engine` |
| 分组方式 | 无分组，边列表扁平 | `_group_by_zone()` 按连通子图分组 |
| `_build_story_prompt()` | per-edge：想法 + 边列表 | per-subgraph：zone节点 + NPC节点(体力/饱腹/心情/记忆/性格/想法) + 库存 + 边 |
| 输出格式 | `【A↔B】` / `【A】` 碎片 | `【场景:xxx】` 连贯场景故事 |
| `_assign_stories()` | 正则匹配每个 `【A↔B】` 分配 | 优先按 `【场景:xxx】` 头部匹配，场景内所有边共享一段故事 |
| `_STORY_HEAD_RE` | 仅匹配 `【A↔B】` / `【A】` | 新增 `_SCENE_HEAD_RE` 匹配 `【场景:xxx】` |

### 3.2 `graph_npc_engine.py`

在 `_execute_intents()` 中为每个 `exec_result` 扩充节点信息：

| 新增字段 | 来源 | 用途 |
|----------|------|------|
| `memories` | NPC model 的 `memory` 列表（最新 4 条） | 让 LLM #3 知道角色近期经历 |
| `mood_text` | 实体 `attributes.mood` → 文字化 | 不用数值暴露，只用自然语言 |
| `satiety_text` | 实体 `attributes.satiety` → 文字化 | 同上 |
| `vitality_text` | 实体 `attributes.vitality` → 文字化 | 同上 |
| `traits` | 实体 `traits` | 性格特质 |

新增模块级辅助函数：
- `_val_to_mood_text(val)` — 数值→文字
- `_val_to_sat_text(val)`
- `_val_to_vit_text(val)`

## 四、改动前后对比

### 输入（LLM #3 prompt）

**前：** 信息贫瘠，只有想法 + 边列表
```
==== 每个 NPC 的想法 ====
【老陈】我叫老陈，我决定用皮毛换点小麦...

==== 本轮交互边 ====
- 老陈@market驻足
- 王老板@market驻足
```

**后：** 每个场景包包含节点全部内容
```
===== 场景: market =====
【场所】market
【角色】老陈（FARMER）
  体力: 精力充沛
  饱腹: 有点饿
  心情: 有点低落
  最近经历: 之前用皮毛换了小麦...
  性格: 平民、普通
  想法: 用皮毛换点金币...
【物品】
  · 老陈带着: 金币x35、皮毛x3
【交互】
  · 老陈 在 market 驻足
```

### 输出（LLM #3 故事）

**前：** 每边碎片
```
【老陈↔market】老陈蹲在石阶上，余光扫到王老板...
【王老板↔market】王老板倚在摊位前盘算着生意...
```

**后：** 每场景连贯叙事
```
【场景:market】清晨的阳光洒在market的石板路上，老陈蹲在摊位前，
王老板在整理货架。两人寒暄几句，默契地开始讨价还价...
```

### 信息流（LLM #3 → LLM #4）

**前：** 故事因头匹配不上 → 被丢弃 → LLM #4 看到"安静地待着"
**后：** 场景故事分给子图内所有边 → LLM #4 看到完整叙事 → 正确推理出 delta

## 五、必要性

### 5.1 不改变，LLM #4 永远缺信息

LLM #4 是**唯一写数值的层**。它写数值的依据是"故事 + 拓扑"。如果故事因格式不匹配被丢弃，LLM #4 只能靠拓扑猜——这就是之前**编造 NPC 不存在的物品**的根因之一。

### 5.2 不改变，叙事与分组矛盾

"每条边一段"的指令鼓励 LLM #3 写多人联合场景（同区域两人），但 `_assign_stories` 按单边匹配。这个矛盾在所有多人在场 tick 都会触发。

### 5.3 不改变，LLM #3 没有写作素材

不给角色状态（心情、记忆力、性格）的说书人，只能反复用"想交易"三字写一千种变体。故事同质化 → LLM #4 判断力下降 → 市场停滞。

---

## 六、优势

### 6.1 拓扑与内容解耦

- 拓扑引擎（`_group_by_zone` / 未来的 BFS）只决定"哪些节点在一起"
- 内容层只提供"每个节点有什么信息"
- LLM #3 只消费内容，从不问"这个分组是怎么来的"

### 6.2 信息不丢失

- 一段场景故事分给所有边 → LLM #4 看到的是完整叙事
- 不再是"安静地待着"这种占位符

### 6.3 叙事更自然

- LLM #3 不用再被"每条边一段"的指令约束，可以自由写多角色互动
- 场景包里的角色记忆、心情、性格给故事提供了真实质地

### 6.4 可扩展

- 未来加新的节点类型（如 `zone` 节点可以加氛围描述），只需在场景包里多写一行
- 分组逻辑从字符串匹配升级为 BFS + `is_leaf` 标签，对拓扑完全透明
