## 1. LLM #4 `craft` 产出格式错误 ⚠️

### 发现时间
Tick 19 (2026-04-29 21:00)

### 现象
RecipeEngine 正确执行了配方（消耗原料、产出物品的 graph edge），但 LLM #4 在 post-processing 中覆写了 inventory_changes，把"这把锄头值 5 金币"的**价值描述**当成了**配方产出**：

```json
// LLM #4 输出（错误）
{"item_name": "金币", "action": "add", "quantity": 5, "type": "craft"}

// 正确应为
{"item_name": "工具", "action": "add", "quantity": 1, "type": "craft"}
```

### 影响范围
- RecipeEngine 产出物丢失（工具/面包等 craft 产物不会正确加入 NPC 库存）
- 消耗端正确（铁锭-2 已正确执行）

### 根因
LLM #4 prompt 中 `craft` 类型的规则只有一行描述：
```python
* "craft"：配方制造（消耗原料产出成品）— 需 recipe 字段标明配方名
```
**没有示例展示 craft 产出应该是物品名（工具/面包），而不是金币值。** LLM #4 读到故事中 "能卖5个金币" 后，把 `金币+5` 当成了产出物。

### 修复方案
在 LLM #4 prompt 中新增 craft 示例：
```python
# 配方制造示例：
# 铁匠王用2个铁锭锻造一把工具（craft）：
#   铁匠王: {"item_name": "铁锭", "action": "remove", "quantity": 2, "type": "craft"},
#           {"item_name": "工具", "action": "add", "quantity": 1, "type": "craft"}
# ⚠️ craft 产出是物理物品（物品名），不是金币等抽象价值
```

### 状态
❌ 待修复

## 2. 卖方主动交易无数据路径 ⚠️

### 发现时间
Tick 15-19 (2026-04-29 19:00-21:30)

### 现象
老陈连续 3 tick 决策 "卖酒给王老板"，但数据层 0 成交。故事层停留在"聊天寒暄"，LLM #4 两侧均无 inventory_changes。

其他类似场景：
- 老张想买干粮/种子 → 故事层写"买了杂粮饼"（T19），但 LLM #4 无 inventory_changes
- 铁匠王"工具卖 5 金币" → LLM #4 错写为 `金币+5 type:craft`

### 根因
系统的 `add + from_npc` inventory 变更范式**只支持接收方输出**。交易的成功模式：

| 场景 | 发起方 | 接收方输出 | 数据层状态 |
|:----|:-----:|:----------:|:--------:|
| 刘猎户卖皮毛 ✅ | 猎户说"卖" | 王老板 `add 皮毛 from_npc 刘猎户` | ✅ |
| 田嫂买蔬菜 ✅ | 田嫂说"买" | 王老板 `remove 蔬菜 to_npc 田嫂` | ✅ |
| 老陈卖酒 ❌ | **老陈说"卖"** | 老陈需要输出 `add 金币 from_npc 王老板`，王老板需要输出 `add 酒 from_npc 老陈` | ❌ 双方都不输出 |

**卖方想卖东西，但在 LLM #1 → LLM #4 之间没有任何管线能将"卖出意图"转化为 inventory_changes。** LLM #4 只从 graph edges 推导变更（谁和谁连接了、谁收到了什么），图的边中不携带"交易方向"信息。

### 影响范围
- 任何卖方发起（非买方发起）的交易都无法在数据层落实
- 目前成功交易全部由**买方**在 LLM #4 输出（王老板作为中心买家）

### 修复方案讨论
**方案 A（推荐）：在 LLM #4 prompt 中明确 "交易由买卖双方共同反映"**
```python
# 示例：老陈卖米酒给王老板，收金币：
#   老陈: {"item_name": "金币", "action": "add", "quantity": 5, "type": "transfer", "from_npc": "王老板"}
#   王老板: {"item_name": "米酒", "action": "add", "quantity": 1, "type": "transfer", "from_npc": "老陈"}
```
也就是说告诉 LLM #4：**两边都只输出"自己收到的"，但收到什么取决于这笔交易中你的角色**。这是 `add + from_npc` 范式的正确应用——老陈收到了金币，王老板收到了酒。

**方案 B（结构性）：修改 LLM #2 意图模型**
在 `interact_with` 中增加 `{"type": "trade", "direction": "sell", "item": "酒", "qty": 2, "target": "王老板"}`。IntentExecutor 根据这个 intent 创建 inventory 变更边，LLM #4 就不再需要推导方向。

### 状态
❌ 待修复（建议先尝试方案 A）

## 3. LLM #4 双向交易方向幻觉 ✅ 已修复

### 发现时间
Tick 15 (2026-04-29 19:00)

### 现象
刘猎户卖 5 皮毛给王老板收 8 金币，LLM #4 数据输出方向完全相反：

```json
// 王老板 add 皮毛x5 from_npc 刘猎户  ✅ 方向正确
// 王老板 add 金币x8 from_npc 刘猎户  ❌ 应为王老板付金币
// 刘猎户 add 皮毛x5 from_npc 王老板  ❌ 应为刘猎户出皮毛
```

### 根因
LLM #4 prompt 的 transfer 规则只给了单向交易示例（老张卖小麦→收金币），无双向示例。第一次出现双向交易（皮毛↔金币）时，LLM #4 在 `from_npc` 方向推导上出错。

### 修复
在 LLM #4 prompt 中新增了双向交易示例：
```python
# 刘猎户卖5皮毛给王老板，收8金币（双向transfer）：
#   王老板: add 皮毛x5 from_npc 刘猎户
#   刘猎户: add 金币x8 from_npc 王老板
# ⚠️ 每个 NPC 只输出自己收到的东西
```

### 验证
Tick 16-19 连续 4 tick 双向交易方向正确 ✅

### 状态
✅ 已验证通过

## 4. 基本需求 vs 性格阈值不一致 🟡

### 发现时间
持续观察（2026-04-29 全程）

### 现象
不同 NPC 对生存需求的响应阈值差异很大：
- 王老板 V=4 才意识到要休息（濒死才动）
- 刘猎户 S=63 就紧张（早早行动）
- 老陈 S=56 才说"饱腹偏低"（偏慢）

这**不是 bug**，而是设计目标：性格决定风险承受能力。但长周期运行时，王老板这样的"冒险型"NPC 反复进入濒死状态（V=4 多次），降低了系统稳定性。

### 影响范围
- 王老板需要手动关注（他容易把自己折腾到濒死）
- 但对其他 NPC 无影响

### 根因
基本需求 prompt 的设计原则是：
> "不写死阈值，只说掉到 0 就出局，性格决定什么时候行动"

这个设计本身正确。王老板的问题是他性格 + 他的商人行为模式（奔波、不吃不喝）产生叠加效果。

### 修复方案
**不修——这是设计预期内的行为。** 王老板 V=4 的极限经历反而让 LLM 生成了更丰富的叙事（"钱再多没命花"的觉醒记忆 0.7）。

### 状态
🟡 设计接受（不修）

## 5. LLM #4 偶尔遗漏 NPC 条目 🟡

### 发现时间
Tick 11 及后续观察

### 现象
LLM #4 有时只输出 3/4 的 NPC update，某个 NPC 被跳过。DB 层面引擎有兜底逻辑，被跳过的 NPC 状态保留上次值，不会丢失。

### 影响范围
- 该 NPC 本次 tick 无属性变更（但故事层可能提到他）
- 下个 tick 恢复显示

### 根因
LLM #4 的 batch 输出能力受 token limit 和上下文长度影响，在出现复杂故事时可能会跳过"没什么事"的 NPC。

### 修复方案
**暂不修。** DB 兜底逻辑正确运作，没有数据丢失。如后续频繁出现（每 tick 都丢）再考虑在 LLM #4 prompt 增加"必须输出所有 6 个 NPC"约束。

### 状态
🟡 观察中（不影响数据完整性）

## 6. 记忆窗口 20 条上限 🟡

### 发现时间
30-tick 夜跑报告

### 现象
NPC 的记忆列表上限 20 条，填满后不再添加新记忆。早期记忆（如"在market醒来"）持续占用窗口。

### 影响范围
- 超过 20 条后新记忆丢失
- 旧记忆保留（含已无关的信息）

### 修复方案
实现记忆压缩/摘要：当记忆数 > 15 时，将 3-5 条低权重（importance < 0.4）的记忆合并为一条摘要。

### 状态

### 修复方案

## 8. 叙事与状态对齐 — LLM #3 需遵守可执行性约束

### 发现时间
2026-04-30 01:20（Tick 1 报告分析）

### 现象
老陈↔王老板皮毛换酒交易中：
- LLM #3 故事：老陈供货皮毛，王老板拿酒换
- DB 事实：王老板持有皮毛×9，老陈持有酒×15
- 故事方向与 DB 状态完全相反
- LLM #4 收到矛盾信号后选错了方向 → 库存数据损坏

### 数据流追踪
```
LLM #1: 老陈计划"用酒换皮毛"                    ✅ 方向正确
        王老板计划"老陈对皮毛感兴趣想进货酒"
                │
                ▼
LLM #3: 故事写"老陈供货皮毛，王老板拿酒换"         ❌ 角色反转
        原因：无结构化库存表，从自然语言模糊推导
                │
                ▼
LLM #4: 看到故事(陈→皮毛) + DB(王有皮毛)          ❌ 矛盾信号
        选了故事方向而非 DB 方向
        输出双向乱 → 自动求净后完全错误
```

### 根因
LLM #3 prompt 里有库存数据（内嵌在 NPC 自然语言计划中），但没有结构化提取方向：
```
王老板: "持有皮毛×9……老陈对皮毛感兴趣想进货酒"
老陈:   "持有酒×15……王老板对皮毛感兴趣想进货酒"
```

LLM #3 关注了自然语言的语义方向（"老陈对皮毛感兴趣" = 老陈是皮毛相关方），
而没有检查数据事实（DB：王老板实际持有皮毛）。

与 #3 双向方向 bug 的区别：
| 维度 | #3 双向方向 bug | #8 叙事状态不对齐 |
|:----|:--------------|:-----------------|
| 故事方向 | ✅ 正确 | ❌ LLM #3 写反 |
| LLM #4 输出 | 格式错 | 方向错（偏离 DB）|
| 根因层级 | LLM #4 prompt 示例不足 | LLM #3 无状态对齐规则 |

### 解决方案
在 LLM #3 prompt 插入一条通用约束：
```
==== 叙事与状态对齐 ====
NPC 只能执行 ta 实际能够执行的行为。
当意图描述中的行动方向与状态数据存在歧义时，
叙事应反映这种不确定性（如未达成/仍在谈/没说清），
而不是强行确定一个可能错误的方向。
```

抽象原则：**叙事层不能输出状态层无法执行的结论。**

### 影响范围
- `interaction_layer.py` — LLM #3 prompt（story layer）追加规则
- 不涉及代码逻辑改动
- 不涉及其他 LLM 层

### 状态
🔄 待实现

## 9. 世界物件与配方工具对象统一 ✅

### 发现时间
2026-04-30 00:40

### 现状
当前系统中有两种 `entity_type="object"` 的创建路径：

| 维度 | 世界物件（18 个） | 配方工具（7 个） |
|:----|:----------------:|:----------------:|
| 创建方式 | `WorldObjectManager` → `object_to_entity()` | `_init_recipe_entities()` 直接 new `Entity()` |
| 类型定义 | `ObjectType` 枚举 | 字符串 |
| 状态机 | ✅ AVAILABLE/OCCUPIED/BROKEN/DEPLETED | ❌ 无 |
| 接口 | interactable + holdable | interactable + holdable（相同） |
| 数量 | 18 个 | 7 个 |

### 根因
实现历史走岔：早期世界物件走 `WorldObject` 基类 → `object_to_entity()`，后期 RecipeEngine 赶工直接 new `Entity()`。两者在图层面完全相同。

### 统一方案
1. `ObjectType` 枚举新增 7 个配方工具值（ANVIL/OVEN/MILL/BREWING_VAT/MORTAR/SEWING_TABLE/WORKBENCH）
2. 新增 `Object` 类（`WorldObject` 子类，`max_concurrency` 替代状态机）
3. `init_default_world()` 末尾统一创建 7 个配方工具
4. `_init_recipe_entities()` 移除手动 Entity 创建（保留 recipe 节点 + NPC 配方绑定）

### 验证
1 tick 通过：RecipeEngine find_interaction_edge("铁砧") 正常，0 crash，grep 零硬编码引用 ✅

### 分类标签
架构改进

### 状态
✅ 已验证通过（2026-04-30 Tick 0）
